# Benji Wave 1 Plan

## Purpose / Big Picture

Wave 1 establishes Benji's product shape inside the existing Craft Agents runtime. The goal is not to finish the full Benji vision. The goal is to make the app feel like a real personal command center with an `Agent` home and a project-centered workbench, while reusing Craft's stable runtime machinery underneath.

Wave 1 should answer one immediate user question:

When Hunter opens the app, does it already feel like Benji is where work lives, review happens, and next actions are launched?

## Surprises & Discoveries

- The current PRD still assumes `Morning Brief` as a top-level nav item, but the desired product direction is now `Agent` as the home surface.
- The most important early win is not deep orchestration. It is changing the user's perception of the app from a session-first cockpit into a command center.
- The repo already has strong note, session, runtime, and panel infrastructure. Wave 1 should wrap and redirect those capabilities before attempting a full data-model rewrite.

## Decision Log

- Decision: Wave 1 home is `Agent`, not `Morning Brief`.
  - Reason: The product should feel like Hunter is stepping into Claw's operating surface, not a generic dashboard.

- Decision: `Project Canvas` is the default project view.
  - Reason: Projects must feel like workbenches, and the document-centered canvas is the clearest place to unify notes, tasks, threads, runs, and later coding.

- Decision: Wave 1 will use adapter layers over existing Craft/session/runtime data where possible.
  - Reason: This gets to visible user value faster than designing the final persistence model first.

- Decision: Artifacts, variants, and embedded coding depth are deferred beyond Wave 1.
  - Reason: The shell and project surfaces need to feel correct before advanced orchestration can land coherently.

## Outcomes & Retrospective

Planned outcomes for Wave 1:

- Opening the app feels like entering Benji, not Craft.
- `Agent` is the default home surface.
- Projects open into a real command-center canvas rather than a session or folder detail page.
- Existing work is visible through Benji concepts, even if some underlying data still comes from Craft sessions.
- The app supports a basic review-and-next-action loop without requiring the full final Benji architecture.

## Plan of Work

### Workstream 1: Align the PRD and product language

Update the PRD to reflect the actual Wave 1 IA and terminology:

- replace `Morning Brief` top-level assumptions with `Agent`
- define `Project Canvas` explicitly as the default project surface
- align language around `threads`, `runs`, and command-center framing

### Workstream 2: Reframe the shell

Replace the visible Craft-first navigation and product language with Benji's Wave 1 model:

- `Agent`
- `Projects`
- `Tasks`
- `Notes`
- `Threads`
- `Queue`
- `Settings`

This work should focus on the visible shell and routing model first, not deep new functionality.

### Workstream 3: Build the Agent home

Create a first-pass `Agent` home surface that acts as the app's command center:

- visible momentum
- pending review items
- blocked work
- active runs
- suggested next actions
- recently active projects

The first version can derive these items from existing runtime/session/note data.

### Workstream 4: Build the Project Canvas shell

Create the default project surface with:

- center living document
- project pulse / activity strip
- right-side context panel
- visible related tasks / threads / runs from the current project context

This is the structural foundation for later artifact review and embedded coding.

### Workstream 5: Add Benji-facing adapters over Craft runtime state

Introduce thin adapter/view-model layers so the UI can reason in Benji nouns even if the backing data is still coming from Craft sessions and other existing structures.

Target concepts:

- `Project`
- `Task`
- `Run`
- `Thread`
- `ReviewItem`

### Workstream 6: Build simple but real list surfaces

Create first-pass views for:

- `Projects`
- `Tasks`
- `Threads`
- `Queue`

These surfaces should be useful enough for navigation and inspection, but they do not need the full final product behavior yet.

## Architecture Impact

Wave 1 should minimize core runtime architecture changes. The main impact is:

- UI shell/routing changes in the renderer
- new Benji-facing view models/adapters over existing runtime/session state
- selective panel and navigation restructuring

`ARCHITECTURE.md` does not need to be updated yet unless implementation reveals a durable new domain layer or persistence model that becomes canonical.

## Concrete Steps

1. Update the PRD to reflect the final Wave 1 information architecture and `Agent` home.
User benefit: planning, tickets, and implementation stop drifting toward a home screen the product no longer wants.

2. Define the Wave 1 routing model and shell states for the new top-level navigation.
User benefit: the app immediately feels like Benji when opened, not like a renamed Craft build.

3. Implement the top-level shell and navigation rewrite.
User benefit: core product orientation becomes obvious without the user having to reinterpret Craft terminology.

4. Build the first-pass `Agent` home using existing runtime/session/project signals.
User benefit: the user opens into momentum, review, and next actions instead of a blank or session-centric surface.

5. Define the `Project Canvas` layout contract and supporting UI regions.
User benefit: every project gets one obvious home for thinking, review, and execution.

6. Implement the `Project Canvas` shell with center document, pulse, and right context panel.
User benefit: projects start to feel like living workbenches rather than collections of disconnected tabs.

7. Add Benji-facing adapter layers for project/task/run/thread/review concepts.
User benefit: the UI can present coherent product concepts now without waiting for a full backend rewrite.

8. Build simple `Projects`, `Tasks`, `Threads`, and `Queue` surfaces on top of those adapters.
User benefit: the user can navigate work through Benji's model and inspect what matters quickly.

9. Add first-pass project-context rendering of existing runs and threads inside the canvas.
User benefit: prior work is visible where it matters instead of being stranded in detached session views.

10. Run a coherence pass to remove remaining high-friction Craft language from Wave 1 surfaces.
User benefit: the product feels intentionally designed for Hunter's workflow rather than partially re-skinned.

## Validation and Acceptance

Wave 1 is successful when:

- the default home is `Agent`
- top-level navigation follows the agreed Wave 1 IA
- opening a project lands in a project canvas, not a session-centric detail view
- the canvas shows a living document plus contextual work around it
- the user can understand what changed, what needs review, and what to do next
- the app feels recognizably more like Benji than Craft even before deeper orchestration features arrive

Validation methods:

- manual walkthrough of all top-level navigation destinations
- screenshot review of shell, Agent home, and Project Canvas
- routing sanity checks for direct navigation and refresh
- smoke test that existing session/runtime functionality still works after shell reframing

## Idempotence and Recovery

- Shell/navigation work should be staged so the current app remains bootable after each ticket.
- Adapter work should be additive first, avoiding destructive migration of existing session/state structures in Wave 1.
- Project Canvas should initially degrade gracefully when contextual task/run/thread data is incomplete.
